# Faza 3.1 — Content & CMS: Blog + Pagini legale + Cookie consent

## Scope

Three independent subsystems delivered together:

1. **Payload CMS** (`apps/cms`) — headless CMS for blog posts and legal/static pages
2. **Storefront content pages** — `/blog`, `/blog/[slug]`, `/blog/categorii/[slug]`, `/pagini/[slug]`, navigation updates
3. **Cookie consent banner** — full GDPR-compliant banner (bar bottom, categories)

---

## Section 1: Payload CMS (`apps/cms`)

### Architecture

Payload CMS v3 runs as a separate Next.js app in the monorepo at `apps/cms`, port **3001**. It connects to the same PostgreSQL instance as Medusa but uses its own table prefix (`cms_`). Turborepo orchestrates all three apps via `pnpm dev` — no changes to `turbo.json` are needed because `pnpm-workspace.yaml` already includes `apps/*`, so any app with `dev`/`build` scripts is picked up automatically.

## Section 2: Storefront — Content Pages

### URL Structure

| URL | Content | revalidate |
|---|---|---|
| `/blog` | All posts listing, client-side category filter | 3600 |
| `/blog/[slug]` | Single article (single column centered) | 3600 |
| `/blog/categorii/[slug]` | Category-filtered listing (SSR, SEO-friendly) | 3600 |
| `/pagini/[slug]` | Static pages from Payload (About, legal) | 3600 |

### Blog listing (`/blog`)

- Grid 3 columns (desktop), 2 columns (tablet), 1 column (mobile)
- Category filter tabs above grid (Toate / Ghiduri / Noutăți / Produse în focus)
- Filter is client-side (all posts fetched at build/revalidate, filtered in browser — no extra API call on tab switch)
- Each card: cover image, category badge, title, excerpt, author, read time, date

### Blog post (`/blog/[slug]`)

- Single column centered layout (max ~700px width)
- Structure: category breadcrumb → title → author + date + read time → cover image → Lexical rich text content
- Below article: "Articole similare" (same category, max 3, exclude current)
- `generateStaticParams` for all published posts at build time

### Category page (`/blog/categorii/[slug]`)

- Same grid layout as `/blog` but pre-filtered by category server-side
- `generateStaticParams` for all categories

### Static pages (`/pagini/[slug]`)

- Minimal layout: title + Lexical rich text content
- `generateStaticParams` for all pages where `showInFooter: true` (the four known legal pages — Despre noi, GDPR, Termeni și condiții, Politica de retur)

### Navigation updates

**Header** — add `Blog` link in main nav alongside `Produse` and `Categorii`.

**Footer** — add `Informații` column with links to pages where `showInFooter: true`:
- Despre noi
- GDPR
- Termeni și condiții
- Politica de retur

Footer links fetched in the Footer Server Component with ISR (`revalidate: 3600`) from Payload `pages` collection filtered by `showInFooter: true`. Payload must be running when the storefront is running; this is a server-side fetch at request time, not a true build-time static value.

### Data fetching

Storefront fetches from Payload REST API (`http://localhost:3001/api`). Pattern consistent with existing Medusa fetches:

```ts
// lib/cms/client.ts — new
const CMS_URL = process.env.PAYLOAD_URL ?? "http://localhost:3001"

export async function getPosts(categorySlug?: string): Promise<Post[]>
export async function getPost(slug: string): Promise<Post | null>
export async function getCategories(): Promise<Category[]>
export async function getPage(slug: string): Promise<Page | null>
export async function getFooterPages(): Promise<Page[]>
```

- All functions use `fetch()` with cache tags for on-demand invalidation:
  - Posts fetches: `{ next: { revalidate: 3600, tags: ["cms-blog"] } }`
  - Pages fetches: `{ next: { revalidate: 3600, tags: ["cms-pages"] } }`
- Only `published` posts returned to storefront (filtered in API query: `?where[status][equals]=published`)

### Lexical rich text rendering

Payload v3 serializes Lexical content as JSON. The storefront uses `@payloadcms/richtext-lexical/react` (re-exported from `@payloadcms/richtext-lexical`) to render this JSON as React components. The `<RichText />` component from that package is used in `post-content.tsx` and the `/pagini/[slug]` page.

**Read time calculation:** `Math.ceil(wordCount / 200)` minutes, where `wordCount` is the approximate word count extracted from the Lexical JSON nodes. Displayed as e.g. "5 min citire".

### On-demand revalidation

The existing `app/api/revalidate/route.ts` already implements tag-based revalidation using `revalidateTag(tag)` with a GET request — **no modification needed**. Payload's `afterChange` hook fires a GET request to the storefront revalidation endpoint:

- On `posts` publish/update: `GET {STOREFRONT_URL}/api/revalidate?secret=REVALIDATE_SECRET&tag=cms-blog`
- On `pages` publish/update: `GET {STOREFRONT_URL}/api/revalidate?secret=REVALIDATE_SECRET&tag=cms-pages`

These tags match the `tags` set in `lib/cms/client.ts` fetch calls, so the ISR cache is invalidated immediately on publish.

Payload hook configured in `payload.config.ts` `afterChange` hooks on `posts` and `pages` collections. `REVALIDATE_SECRET` is the shared secret already present in both apps' env vars.

## Section 3: Cookie Consent Banner

### Implementation

`vanilla-cookieconsent` v3 — zero React dependencies, 4kb, full GDPR.

Implemented as a `"use client"` component `<CookieConsent />` imported and rendered in `app/layout.tsx`. This is valid Next.js App Router usage — Server Components (like `layout.tsx`) can import and render Client Components. The `"use client"` boundary means only `cookie-consent.tsx` and its imports run on the client.

### Categories

| Category | Key | Default | Description |
|---|---|---|---|
| Necesare | `necessary` | Always on | Session, cart, auth cookies |
| Analitice | `analytics` | Off | Future: traffic analytics |
| Marketing | `marketing` | Off | Future: ad retargeting |

### UX

- **Position:** Full-width bar at bottom of viewport (does not block content)
- **First visit:** Bar appears with two buttons: "Acceptă toate" + "Personalizează"
- **"Personalizează"** expands panel with category toggles (Necesare always locked on)
- **Preference stored** in cookie `cc_cookie` (1-year expiry)
- **Re-consent** triggered after 365 days or if categories change
- **Language:** Romanian throughout

### Component structure

```
components/cookie-consent/
└── cookie-consent.tsx    ← "use client", initializes vanilla-cookieconsent
```

## Section 4: Environment Variables

**`apps/cms/.env`** (local, not committed) + **`apps/cms/.env.example`** (committed):

```
DATABASE_URI=postgresql://user:pass@localhost:5432/firintins
PAYLOAD_SECRET=<random 32+ char string>
NEXT_PUBLIC_SERVER_URL=http://localhost:3001
```

**`apps/storefront/.env.local`** (additions — document in `.env.example` for storefront if one exists):

```
PAYLOAD_URL=http://localhost:3001
REVALIDATE_SECRET=<shared with Payload — must match value in apps/cms/.env>
```

`REVALIDATE_SECRET` is already present in the storefront env (used by the existing revalidation route). The same value must be set in `apps/cms/.env` so the Payload afterChange hook can authenticate its GET request to `/api/revalidate`.
