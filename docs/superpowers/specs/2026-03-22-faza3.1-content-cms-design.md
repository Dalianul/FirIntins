# Faza 3.1 — Content & CMS: Blog + Pagini legale + Cookie consent

## Scope

Three independent subsystems delivered together:

1. **Payload CMS** — embedded in `apps/storefront`, headless CMS for blog posts and legal/static pages
2. **Storefront content pages** — `/blog`, `/blog/[slug]`, `/blog/categorii/[slug]`, `/pagini/[slug]`, navigation updates
3. **Cookie consent banner** — full GDPR-compliant banner (bar bottom, categories)

---

## Section 1: Payload CMS (embedded in `apps/storefront`)

### Architecture

Payload CMS v3 runs **embedded inside the existing storefront Next.js app** — not as a separate process or port. This follows the official Medusa + Payload integration pattern. The Payload admin UI is served at `http://localhost:3000/admin`, and blog/pages data is accessed via **Payload's local API** (direct database queries, no HTTP) from Server Components.

Benefits over a separate `apps/cms`:
- No inter-service HTTP calls — Server Components query the DB directly via `getPayload({ config })`
- In-process cache revalidation — `afterChange` hooks call `revalidateTag()` directly
- Single Next.js process, consistent Next.js version (16.x)
- Follows the Medusa-recommended integration approach

**Next.js compatibility:** `@payloadcms/next` is compatible with Next.js 16.x (storefront uses `16.2.1`).

**Database:** Payload uses a **separate PostgreSQL database** (not the same database as Medusa). This avoids any risk of table-level collision. Connection string in `PAYLOAD_DATABASE_URL`.

## Section 2: Storefront — Content Pages

### URL Structure

| URL | Content | revalidate |
|---|---|---|
| `/blog` | All posts listing, client-side category filter | 3600 |
| `/blog/[slug]` | Single article (single column centered) | 3600 |
| `/blog/categorii/[slug]` | Category-filtered listing (SSR, SEO-friendly) | 3600 |
| `/pagini/[slug]` | Static pages from Payload (About, legal) | 3600 |

### Blog listing (`/blog`)

- Grid 3 columns (desktop), 2 columns (tablet), 1 column (mobile)
- Category filter tabs above grid (Toate / Ghiduri / Noutăți / Produse în focus)
- Filter is client-side (all posts fetched server-side, filtered in browser — no extra API call on tab switch)
- Each card: cover image, category badge, title, excerpt, author, read time, date

### Blog post (`/blog/[slug]`)

- Single column centered layout (max ~700px width)
- Structure: category breadcrumb → title → author + date + read time → cover image → Lexical rich text content
- Below article: "Articole similare" (same category, max 3, exclude current)
- `generateStaticParams` for all published posts at build time

### Category page (`/blog/categorii/[slug]`)

- Same grid layout as `/blog` but pre-filtered by category server-side
- `generateStaticParams` for all categories

### Static pages (`/pagini/[slug]`)

- Minimal layout: title + Lexical rich text content
- `generateStaticParams` for all pages where `showInFooter: true` (the four known legal pages: Despre noi, GDPR, Termeni și condiții, Politica de retur)

### Navigation updates

**Header** — add `Blog` link in main nav alongside `Produse` and `Categorii`.

**Footer** — add `Informații` column with links to pages where `showInFooter: true`:
- Despre noi
- GDPR
- Termeni și condiții
- Politica de retur

Footer links fetched in the Footer Server Component via `getCachedFooterPages()` (Payload local API, `'use cache'` with tag `"cms-pages"`, revalidate 3600s). If the database is unavailable, the inner `getFooterPages()` catches and returns an empty array (footer renders without the Informații column).

### Data fetching — Payload Local API

Server Components use the Payload local API (direct DB access, no HTTP). This is more efficient than REST fetches and eliminates the inter-service network dependency.

```ts
// lib/cms/client.ts
import { getPayload } from "payload"
import config from "@payload-config"

export async function getPosts(categorySlug?: string) {
  const payload = await getPayload({ config })
  const { docs } = await payload.find({
    collection: "posts",
    where: {
      status: { equals: "published" },
      ...(categorySlug ? { "category.slug": { equals: categorySlug } } : {}),
    },
    depth: 1,         // resolves category relationship
    pagination: false,
  })
  return docs
}

export async function getPost(slug: string) {
  const payload = await getPayload({ config })
  const { docs } = await payload.find({
    collection: "posts",
    where: { slug: { equals: slug }, status: { equals: "published" } },
    depth: 1,
    limit: 1,
  })
  return docs[0] ?? null
}

export async function getCategories() {
  const payload = await getPayload({ config })
  const { docs } = await payload.find({ collection: "categories", pagination: false })
  return docs
}

export async function getPage(slug: string) {
  const payload = await getPayload({ config })
  const { docs } = await payload.find({
    collection: "pages",
    where: { slug: { equals: slug } },
    limit: 1,
  })
  return docs[0] ?? null
}

export async function getFooterPages() {
  try {
    const payload = await getPayload({ config })
    const { docs } = await payload.find({
      collection: "pages",
      where: { showInFooter: { equals: true } },
      pagination: false,
    })
    return docs
  } catch {
    return []
  }
}
```

**Cache tagging for ISR:** Use the `'use cache'` directive with `cacheTag()` and `cacheLife()` — the Next.js 16 replacement for deprecated `unstable_cache`. Requires `cacheComponents: true` in `next.config.ts` (set alongside `withPayload()`):

```ts
// lib/cms/client.ts — cached wrappers
import { cacheTag, cacheLife } from "next/cache"

export async function getCachedPosts(categorySlug?: string) {
  "use cache"
  cacheTag("cms-blog")
  cacheLife({ revalidate: 3600 })
  return getPosts(categorySlug)
}

export async function getCachedFooterPages() {
  "use cache"
  cacheTag("cms-pages")
  cacheLife({ revalidate: 3600 })
  return getFooterPages()
}
```

**`next.config.ts`** must enable `cacheComponents`:

```ts
import withPayload from "@payloadcms/next/withPayload"

const nextConfig = {
  cacheComponents: true,
  // ...existing config
}

export default withPayload(nextConfig)
```

### Lexical rich text rendering

Payload v3 serializes Lexical content as JSON. Use `@payloadcms/richtext-lexical/react`'s `<RichText />` component in `components/blog/post-content.tsx` and the `/pagini/[slug]` page.

**Read time calculation:** `Math.max(1, Math.ceil(wordCount / 200))` minutes, where `wordCount` is extracted by walking Lexical JSON `text` leaf nodes and splitting on whitespace. Displayed as e.g. "5 min citire".

### On-demand revalidation

Payload `afterChange` hooks in `payload.config.ts` call `revalidateTag()` directly — no HTTP webhook needed since Payload runs in the same Next.js process. In embedded mode, admin UI operations trigger `afterChange` within a request context, so `revalidateTag()` works. Wrap defensively for edge cases (CLI imports, seed scripts):

```ts
// payload.config.ts afterChange hooks
import { revalidateTag } from "next/cache"

// Posts collection afterChange
afterChange: () => {
  try { revalidateTag("cms-blog") } catch (e) { console.warn("cms-blog revalidation skipped:", e) }
}

// Pages collection afterChange
afterChange: () => {
  try { revalidateTag("cms-pages") } catch (e) { console.warn("cms-pages revalidation skipped:", e) }
}
```

If running bulk imports or seed scripts outside the Next.js server, cache revalidation will not fire — an explicit request to `/api/revalidate` is the fallback for those cases.

The existing `app/api/revalidate/route.ts` requires **no changes** (it's still available for external triggers if needed, but Payload no longer calls it).

## Section 3: Cookie Consent Banner

### Implementation

`vanilla-cookieconsent` v3 — zero React dependencies, 4kb, full GDPR. Added to storefront `package.json`.

Implemented as a `"use client"` component `<CookieConsent />` imported and rendered in `app/layout.tsx`. This is valid Next.js App Router usage — Server Components can import and render Client Components. The `"use client"` boundary means only `cookie-consent.tsx` and its imports run on the client.

### Initialization pattern

```ts
// components/cookie-consent/cookie-consent.tsx
"use client"
import { useEffect } from "react"
import CookieConsentLib from "vanilla-cookieconsent"
import "vanilla-cookieconsent/dist/cookieconsent.css"

export function CookieConsent() {
  useEffect(() => {
    CookieConsentLib.run({
      guiOptions: { consentModal: { layout: "bar", position: "bottom" } },
      categories: {
        necessary: { enabled: true, readOnly: true },
        analytics: { enabled: false },
        marketing: { enabled: false },
      },
      language: {
        default: "ro",
        translations: {
          ro: {
            consentModal: {
              title: "Setări cookie-uri",
              description: "Folosim cookie-uri esențiale, analitice și de marketing.",
              acceptAllBtn: "Acceptă toate",
              showPreferencesBtn: "Personalizează",
            },
            preferencesModal: {
              title: "Preferințe cookie-uri",
              savePreferencesBtn: "Salvează preferințele",
              sections: [
                { title: "Necesare", description: "Session, coș, autentificare.", linkedCategory: "necessary" },
                { title: "Analitice", description: "Trafic anonim.", linkedCategory: "analytics" },
                { title: "Marketing", description: "Reclame personalizate.", linkedCategory: "marketing" },
              ],
            },
          },
        },
      },
    })
  }, [])
  return null
}
```

### Categories

| Category | Key | Default | Description |
|---|---|---|---|
| Necesare | `necessary` | Always on | Session, cart, auth cookies |
| Analitice | `analytics` | Off | Future: traffic analytics |
| Marketing | `marketing` | Off | Future: ad retargeting |

### UX

- **Position:** Full-width bar at bottom of viewport (does not block content)
- **First visit:** Bar appears with two buttons: "Acceptă toate" + "Personalizează"
- **"Personalizează"** expands panel with category toggles (Necesare always locked on)
- **Preference stored** in cookie `cc_cookie` (1-year expiry)
- **Re-consent** triggered after 365 days or if categories change
- **Language:** Romanian throughout

```

└── cookie-consent.tsx    ← "use client", initializes vanilla-cookieconsent
```

## Section 5: Middleware Update

The existing `middleware.ts` uses `matcher: ["/cont/:path*"]` — it only runs on `/cont` routes. Payload routes (`/admin`, `/api/payload`) are **already excluded** by the matcher and require **no code changes**.

**No changes needed to `middleware.ts`.** Verify after installing Payload that the matcher has not been modified and Payload admin routes remain outside the matcher scope. If the matcher is ever broadened to cover all routes, add a Payload exclusion check:

```ts
// Only needed if matcher is changed to cover all routes:
if (pathname.startsWith('/admin') || pathname.startsWith('/api/payload')) {
  return NextResponse.next()
}
```
