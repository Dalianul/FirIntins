# Faza 3.3 — Checkout Emails & Stripe Webhooks — Design Spec

**Date:** 2026-03-30
**Status:** Approved
**Scope:** Backend (Medusa v2 subscribers + SendGrid Dynamic Templates) + Stripe webhook configuration

---

## Context

The checkout flow (address → shipping → Stripe payment → order confirmation) is fully functional. However:

- `src/subscribers/` is empty — no events trigger emails
- SendGrid is configured in `medusa-config.ts` but wired to nothing
- The confirmation page already shows "Ți-am trimis o confirmare pe email" — currently false
- `STRIPE_WEBHOOK_SECRET` is configured but the Stripe webhook endpoint is not registered in the Stripe Dashboard

This spec covers making all three gaps production-ready.

---

## Architecture

### Approach: 3 independent subscribers (Medusa v2 pattern)

One file per event. Each subscriber is self-contained: resolves container dependencies, fetches order data via `query.graph()`, sends notification via `notificationModuleService.createNotifications()`.

No shared helpers or abstractions — follows the exact pattern from Medusa v2 docs. Refactor if/when the number of email types grows beyond ~5.

### Files to create

```
apps/backend/src/subscribers/
  order-placed.ts              ← order.placed event
  order-fulfillment-created.ts ← order.fulfillment_created event
  order-canceled.ts            ← order.canceled event
```

### Env vars to add

```bash
# apps/backend/.env (and .env.template)
SENDGRID_TEMPLATE_ORDER_PLACED=d-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
SENDGRID_TEMPLATE_ORDER_SHIPPED=d-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
SENDGRID_TEMPLATE_ORDER_CANCELED=d-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

Template IDs come from SendGrid Dashboard → Email API → Dynamic Templates. Each must be created manually in SendGrid with the corresponding variable names from the data payloads below.

## Subscriber 1: `order-placed.ts`

**Event:** `order.placed`
**Payload:** `{ id: string }` (order ID)

### Fields fetched via `query.graph()`

```
id, display_id, email, created_at, currency_code,
items.*, items.variant.*, items.variant.product.*,
shipping_address.*,
shipping_methods.*, shipping_methods.shipping_option.*,
summary.*,
customer.*
```

### Data sent to SendGrid template

```ts
{
  order_id: order.display_id,
  customer_name: order.shipping_address?.first_name ?? order.email,
  email: order.email,
  items: order.items?.map(item => ({
    title: item.title,
    thumbnail: item.thumbnail,
    quantity: item.quantity,
    unit_price: item.unit_price,
    subtotal: item.subtotal,
  })),
  shipping_address: {
    address_1: order.shipping_address?.address_1,
    city: order.shipping_address?.city,
    postal_code: order.shipping_address?.postal_code,
  },
  subtotal: order.summary?.raw_original_order_total?.value,
  shipping_total: order.summary?.shipping_total,
  tax_total: order.summary?.tax_total,
  total: order.summary?.raw_current_order_total?.value,
  currency_code: order.currency_code,
}
```

**Event:** `order.fulfillment_created`
**Payload:** `{ id: string }` (order ID)

### Fields fetched

```

shipping_address.*,

```

### Data sent to SendGrid template

```ts
{
  order_id: order.display_id,
  customer_name: order.shipping_address?.first_name ?? order.email,

    postal_code: order.shipping_address?.postal_code,
```

## Error Handling

Each subscriber wraps its logic in `try/catch`:

- **Missing template ID:** If the env var is not set, log a warning and return early. Do not throw.
- **SendGrid failure:** Catch and log the error. Do not throw — email failure must never block order processing.
- **Missing order data:** If `query.graph()` returns no results, log and return.

```ts
const templateId = process.env.SENDGRID_TEMPLATE_ORDER_PLACED
if (!templateId) {
  console.warn("[order-placed] SENDGRID_TEMPLATE_ORDER_PLACED not set, skipping email")
  return
}

try {
  await notificationModuleService.createNotifications({ ... })
} catch (error) {
  console.error(`[order-placed] Failed to send email for order ${data.id}:`, error)
}
```

## Stripe Webhook Configuration

No code changes required. Medusa v2's `@medusajs/payment-stripe` module handles Stripe webhook events internally at the endpoint:

```
POST <MEDUSA_BACKEND_URL>/hooks/payment/stripe
```

**Setup steps (manual, not code):**
1. Open Stripe Dashboard → Developers → Webhooks
2. Add endpoint: `https://api.firintins.ro/hooks/payment/stripe`
3. Select events: `payment_intent.succeeded`, `payment_intent.payment_failed`
4. Copy the generated webhook signing secret
5. Set `STRIPE_WEBHOOK_SECRET=whsec_...` in `apps/backend/.env`

For local development, use Stripe CLI:
```bash
stripe listen --forward-to localhost:9000/hooks/payment/stripe
```

## Testing

No automated unit tests for subscribers (they are thin wrappers over Medusa SDK; meaningful testing requires an integration environment).

**Manual smoke test checklist:**
- [ ] Place a test order in storefront → email arrives in inbox, correct items/totals/address
- [ ] Create fulfillment in Medusa Admin for that order → "expediată" email arrives
- [ ] Cancel an order from Medusa Admin → cancellation email arrives
- [ ] Remove `SENDGRID_TEMPLATE_ORDER_PLACED` from `.env` → warning logged, no crash
- [ ] Stripe webhook: `stripe listen --forward-to localhost:9000/hooks/payment/stripe` → place order → webhook received with 200
