# Faza 2.3 — Cont complet: Profil, Adrese, Securitate

## Scope

Three new account sub-pages under `/cont/`:
- `/cont/profil` — edit first name, last name, email
- `/cont/adrese` — manage saved addresses (add, edit, delete, set default)
- `/cont/securitate` — change password

Plus: update `AccountNav` sidebar with the three new links and move logout button into the sidebar.

---

## Section 1: Layout & Navigation

### Two-column layout (desktop)

The account section uses a persistent two-column grid:
- **Left sidebar** (~220px): `AccountNav` component with all nav links + logout button at the bottom
- **Right content** (flex-1): page-specific content

Mobile collapses to a horizontal scrollable tab bar above the content area.

### AccountNav updates

`components/account/account-nav.tsx` gains three new nav links:

| Label | Path |
|---|---|
| Tablou de bord | `/cont` |
| Comenzile mele | `/cont/comenzi` |
| Lista de dorinte | `/cont/dorinte` |
| Adresele mele | `/cont/adrese` |
| Profilul meu | `/cont/profil` |
| Securitate | `/cont/securitate` |

Logout button moves from the dashboard page content into the sidebar bottom.

### Layout component

`app/cont/layout.tsx` currently has a single-column centered layout. It is replaced with a two-column grid that renders `<AccountNav />` once for all child pages:

```tsx
<div className="max-w-7xl mx-auto px-6 py-12">
  <div className="flex flex-col md:flex-row gap-8">
    <AccountNav />
    <div className="flex-1 min-w-0">{children}</div>
  </div>
</div>
```

`app/cont/page.tsx` currently renders `<AccountNav />` inline in its content. Since the layout now owns the nav, the dashboard page must remove its `<AccountNav />` render to prevent duplicate nav. The logout button is also removed from the dashboard page (it moves into `AccountNav`).

## Section 2: `/cont/profil` — Editare profil

### Fields
- `firstName` (required)
- `lastName` (required)
- `email` (required, valid email format)

### Zod schema
`lib/schema/account.ts` — `profileSchema`

### Server Action
`actions/account.ts` — `updateProfileAction`
- Calls `sdk.store.customer.update({ first_name, last_name, email })`
- On success: `revalidatePath("/cont/profil")`, returns `{ success: true }`
- On error: returns `{ error: string }`

### UX
- Form is always editable (no view/edit toggle)
- Success: page re-renders with inline message via `?success=1` search param
- Error: field-level Zod errors displayed under each input
- Submit button disabled while pending

## Section 3: `/cont/adrese` — Gestionare adrese

### Address card
Each saved address displays:
- Full name, street (address1 + optional address2), city, province, postal code, country
- "Implicita" badge on the default address
- Edit icon button → opens Sheet pre-filled
- Delete button → disabled on default address (tooltip: "Seteaza alta adresa ca implicita inainte de a sterge")
- "Seteaza ca implicita" button on non-default cards

### Address Sheet (shadcn `Sheet`)
Slides from the right. Used for both add and edit.

**Fields:**
- `firstName`, `lastName` (required)
- `address1` (required), `address2` (optional)
- `city`, `province`, `postalCode` (required)
- `countryCode` (required, select)
- `phone` (optional)

### Zod schema
`lib/schema/account.ts` — `addressSchema`

### Server Actions (`actions/account.ts`)
- `addAddressAction` → `sdk.store.customer.createAddress()`
- `updateAddressAction` → `sdk.store.customer.updateAddress()`
- `deleteAddressAction` → `sdk.store.customer.deleteAddress()`
- `setDefaultAddressAction` → `sdk.store.customer.updateAddress(addressId, { is_default_shipping: true })` — Medusa atomically unsets any previously default address, so only a single API call is needed

All actions call `revalidatePath("/cont/adrese")` on success.

## Section 4: `/cont/securitate` — Securitate

### Fields
- `password` (required, min 8 chars)
- `confirmPassword` (required, must match `password`)

### Zod schema
`lib/schema/account.ts` — add new export `passwordSchema` with `z.refine` for match validation. This is distinct from the existing `changePasswordSchema` (which includes `currentPassword`); the new schema has only `password` + `confirmPassword` since Medusa's store customer update API does not require the current password.

### Server Action
`actions/account.ts` — `updatePasswordAction`
- Calls `sdk.store.customer.update({ password })`
- On success: `revalidatePath("/cont/securitate")`, returns `{ success: true }`
- On error: returns `{ error: string }`

### UX
- Form resets after successful update
- Inline success message ("Parola a fost actualizata.")
- No current password field (not required by Medusa store API)

---

## Section 5: Data & Caching

- **Customer data**: fetched server-side in each page via `sdk.store.customer.retrieve()` with session cookie from `cookies()`
- **No React Query**: pure Server Components + Server Actions — data always fresh on navigation
- **Revalidation**: each Server Action calls `revalidatePath()` for the relevant path after mutation
- **Redirect**: profile and security forms redirect to the same page with `?success=1` after success; address Sheet closes and the page revalidates
- **Session guard**: if `sdk.store.customer.retrieve()` fails, redirect to `/autentificare`
